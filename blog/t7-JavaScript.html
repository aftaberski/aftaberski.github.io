Face Off! Ruby Classes v. JavaScript Constructor Functions

DECEMBER 14, 2014

We've ventured away from Ruby and started learning a bit about JavaScript. Turns out, they have a decent number of similarities but with a few nuances. Here we'll take a look at Ruby classes and JavaScript constructor functions & prototypes.

A Bit About Constructor Functions & Prototypes

In JavaScript, Contructor functions...well, functions, which in turn are objects. They can be combined with the new keyword to create a new object and initialize it with a prototype. For example:

function HolidayCharacter() {
  this.name = "Santa";
  this.holiday = "Christmas";
}

HolidayCharacter.prototype.print = function(){
  console.log(this.name + " is associated with " + this.holiday + "!");
}

var santa = new HolidayCharacter();
santa.print();
//=> Santa is associated with Christmas

The above function is a constructor function because it's purpose is to create new objects. Why not just use the literal definitions? Why complicate it with all this? Like classes, constructor functions allow us to define behavior that will be shared by all new objects that are created by that function. This is due to the Function.prototype property.

For example, in the code above, every object created from HolidayCharacter will inherit the print method function.

The Breakdown

Ruby uses objects inherited from a class to create new objects. Similarly, JavaScript lets us define literal objects or use constructor functions to create new objects.

However, JavaScript lets us create new objects with inherited behaviors usings the Function.prototype. In Ruby inheritance is achieved through class objects and method definition. Additionally, in Ruby all classes are instances of the Class metaclasses. JavaScript does not have a metaclass equivalent.
